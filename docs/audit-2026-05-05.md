# Lucid — End-to-End Audit (2026-05-05)

> Comprehensive audit of the project's current state. Scope: every dimension that affects whether Lucid works, ships, attracts contributors, and credibly positions itself as the fluency standard.

**Overall grade: A−**

The architecture, evals, and core code are A-grade. Documentation drift is the single largest issue and drags the overall grade. Project is shippable today; one focused doc-cleanup pass would lift it to A.

---

## 1. What Lucid is (in a paragraph)

Lucid is **the fluency layer between human intent and AI output** — middleware that intercepts vague user requests, extracts true intent (including unspoken constraints), translates intent into the prompt shape the underlying model performs best on, and grades the output against a per-domain rubric before returning. Built on the Anthropic API and shipped as both a standalone Claude skill (zero-setup, recommended for most users) and a Model Context Protocol (MCP) server (advanced install with separate Listener / Translator / Validator pipeline). **Measured win rate: 79.4% decisive vs. baseline raw prompt on the same model, n=51, position-debiased Opus 4.6 judge.**

Thesis pull-quote: *"The dominant constraint on real-world AI usefulness is not model capability. It is the loss of fidelity in the channel between the human and the model. Lucid closes that channel."*

`lucid-check` provides a no-API-needed health check that confirms registry load, triage, and pipeline integrity for all four install paths.

## 3. What Lucid does — four-layer protocol + 11 verticals

### Layers

1. **Listener** (Haiku 4.5) — extracts answers to per-vertical questions from the raw intent via Anthropic forced tool-use. Degrades gracefully on API failure.
2. **Translator** (Sonnet 4.6) — renders the matched vertical's `prompt_template` with extracted answers, sends to the execution model. System prompt cached via `cache_control: ephemeral`.
3. **Validator** (Sonnet 4.6, opt-in via `validate=True`) — grades the output against the vertical's rubric (0.0–1.0 per criterion, weighted average vs. `pass_threshold`). One-rerun budget if score is below threshold.
4. **Memory** (Phase 5, not yet built) — per-user persistent context. Currently a roadmap item, not a feature.

### Verticals (11 total: 10 specialized + 1 universal fallback)

| Vertical | Domain | Status |
|---|---|---|
| `document.one_pager` | One-pagers, executive summaries, decision memos | A — best-performing (87% decisive win rate) |
| `email.professional` | Email drafting/rewriting | A |
| `marketing.copy` | Ads, landing, slogans, value props | B+ — 1 specificity miss in eval |
| `social.thread` | X / LinkedIn / Instagram / Threads / TikTok | B — all ties (3/3) — neither winning nor losing meaningfully |
| `creative.story` | Fiction, scenes, ad scripts | A |
| `code.review` | Refactor / review / explain | C — 0 W / 1 L / 1 T in eval |
| `explain.feynman` | Two-level dual-audience explanation | A− |
| `explain.socratic` | Single-question dialogue starter (multi-turn-by-design) | B − 1 specificity miss in eval |
| `analysis.recommendation` | Strategic recommendations from inputs | A |
| `prompt.image` | Image-gen prompts (Midjourney / Flux / DALL·E / SDXL / Ideogram / Imagen / model-agnostic) | C — 1 W / 2 L in eval |
| `general.fluency` | Universal fallback when no specialized vertical matches | A — never blocks |

### Triage routing

Keyword-substring matching with priority tiebreakers and an `is_fallback` field for `general.fluency`. Verified clean across 13 test intents during the build sequence (no false positives, no domain crossover bugs after the marketing/social keyword fix in commit 61c044c).

## 4. Audit by domain — graded

### A. Architecture & Code Quality · **A**

**What's strong:**
- Clean module separation: `server.py`, `listener.py`, `translator.py`, `validator.py`, `check.py`, `verticals/_schema.py`, `verticals/_loader.py`. Each is focused, well-typed, well-docstringed.
- Dependency injection throughout — every layer accepts `client=None` for stub mode, making the entire pipeline testable without an API key.
- Pydantic v2 schema with cross-field validators (placeholder validation against question ids, fallback uniqueness).
- Anthropic prompt caching (`cache_control: ephemeral`) on Translator system prompts and Listener/Validator tool definitions.
- Error handling at every API boundary — exceptions are caught and surfaced via `.error` fields rather than crashing the pipeline.
- 102 tests, clean separation by module, fast (< 2.5s).

**What could be better:**
- No type-checker config (`mypy` / `pyright`) wired in CI. Relies on developer discipline.
- No coverage measurement. We don't know what's untested.
- No async paths — `mcp.run()` is sync. Fine for v0.2 but limits throughput on busy plugin installs.

### B. Verticals · **A−**

**What's strong:**
- Schema-first design: every vertical is a pure YAML file. No code in verticals.
- Universal fallback (`general.fluency`) means there's no "no_match" failure path.
- Loader filters underscore-prefixed directories (`_template/`) so contributors copy from a clear scaffold rather than from a real vertical.
- Triage tested clean across all 11 verticals + cross-domain routing checks.

**What could be better:**
- DRY violation: `audience` and `constraints` questions appear in 3-4 verticals with slightly different `why_it_matters` text. Tracked as punch-list item #25 (BACKLOG, awaiting trigger conditions).
- `code.review` and `prompt.image` verticals are losing in evals — not a vertical-architecture problem but a vertical-content problem. Need iteration on the prompt templates or rubric weights.
- No version-bump discipline on individual vertical YAML files. All sit at `version: 0.1.0`. Worth defining when a vertical earns a version bump.

### C. Skill (lucid-fluency) · **A**

**What's strong:**
- Standalone (`skill/`) and plugin-bundled (`plugin/skills/`) versions are byte-identical (synced via `cp` on every change).
- v3 iteration based on real eval data — fixed the v1 "asks too aggressively" bug → over-corrected in v2 (bracketed placeholders) → corrected to v3 (illustrative specifics). Full evidence trail in `evals/results/`.
- Anti-pattern guards explicit by name (no "I hope this email finds you well", no "🧵 Thread!", no "Beautiful, stunning, high quality" generics).
- Internalized seven canonical principles from `docs/principles.md`.

**What could be better:**
- Skill description (`description:` in YAML frontmatter) is a long paragraph. Anthropic's auto-trigger uses this for matching — could potentially over-trigger on casual questions. Not yet observed as a problem but worth monitoring.
- Skill body is 100% behavioral instructions, no examples. A few "before/after" examples baked in might help model adherence.

### D. Plugin · **B+**

**What's strong:**
- Manifest, MCP config, slash command, and skill all bundled correctly.
- `.claude-plugin/plugin.json` has clean metadata.
- `commands/lucid.md` slash command file is precise about how to handle each `lucid_run` response status.

**What's stale or broken:**
- ❗ **`plugin.json` version says `"0.1.0"`** — should be `0.2.0` to match `pyproject.toml` and `src/lucid/__init__.py`. Tag release is at v0.2.1.
- `plugin.json` `homepage` and `repository` still point at `https://github.com/Jonnycatx/lucid` — fine until the org rename, then needs update.

### F. Eval & Measurement · **A**

**What's strong:**
- 51 prompts spanning 14 domains.
- Position-debiased Opus 4.6 judging (each pair judged twice with positions swapped; disagreements → tie).
- Both stub-mode and live API paths in the harness, separately tested.
- `--ids` flag (added today) for curated cross-vertical sampling.
- All eval results saved as JSON in `evals/results/`, force-added past gitignore as evidence trail (4 files now).
- v1 → v3 skill iteration was eval-driven, with judge reasoning preserved.

**What could be better:**
- No statistical significance reporting. We say "79.4% decisive on n=51" but no confidence interval. At n=51 with this distribution, the 95% CI is roughly ±10 points — should state that.
- No 2-turn eval mode (punch-list item #5, NEXT). The `explain.socratic` vertical's design is multi-turn but the eval treats it one-shot. Currently we judge first-question quality only.
- No CI eval gate (item #17 PARKED). Nothing prevents a contributor from regressing the win rate.
- One brittle test was found and fixed (`test_prompts_yaml_loads_30_entries`). Worth a sweep for similar hardcoded counts.

## 7. Verdict

**Lucid is shippable today.**

- Architecture, code quality, evals, demo site, skill behavior — all A-grade.
- Documentation is the one drag — three files (`plan.md`, `evals/README.md`, `CONTRIBUTING.md`) and one manifest (`plugin.json`) are stale relative to reality. **One focused 30-minute commit fixes them all.**

The project went from "interesting idea" to "measured, reproducible, contributor-ready open-source project with published evidence" in this session. The substance is real: 79.4% decisive win rate is a defensible headline number, the 11 verticals each have an articulated design and rubric, the eval discipline is real, and the skill iteration was data-driven.

The biggest risk to credibility is **the contradiction between the project's actual state (mature, measured, complete) and its documentation's claimed state (alpha, eval being built, Phase 2 in progress)**. A reader who reads the README sees v0.2 with measured numbers; a reader who clicks into `plan.md` or `CONTRIBUTING.md` sees a project that hasn't shipped Phase 2 yet. **Fix the docs first; everything else can wait.**
